# Trial Workflow (design)

A guided, config-driven flow that walks a rep through helping a customer start a
free trial. Chosen approach: **Option B — config-driven.** AI generates a
structured *workflow config* (from a Google Doc or other source); **one fixed
renderer** displays it. AI never generates UI or code.

> Status: **design doc**. Not yet implemented. Documented so the build stays
> consistent with the app's "data, not code" and UI-stability principles
> ([UI_RULES.md](./UI_RULES.md), [CHANGE_CONTROL.md](./CHANGE_CONTROL.md)).

## Why config-driven (not AI-generated UI)

The whole point of the docs charter is **no unpredictable UI changes**. So the
*flexibility* lives in the data (the steps), and the *presentation* is a single
reviewed component. Want richer layouts later? Add new typed **block types** to
the schema — a controlled vocabulary — never freeform generated UI.

## Proposed schema

A typed config, validated like everything else. When implemented, add these to
`src/types/index.ts` and `playbook.schema.json` in the same commit
([CHANGE_CONTROL.md](./CHANGE_CONTROL.md)).

```ts
interface TrialStep {
  id: string                 // stable id (see merge rules in CONTENT_PIPELINE.md)
  title: string              // e.g. "Create the trial account"
  instruction: string        // what the rep does / guides the customer to do
  sayThis?: string           // customer-facing language
  link?: { label: string; url: string }   // e.g. signup page, docs
  checklistItems?: string[]  // sub-steps the rep ticks off
  estimatedMinutes?: number
}

interface TrialWorkflow {
  id: string
  name: string               // e.g. "Standard 14-day trial"
  industries?: string[]      // optional targeting, same semantics as content items
  steps: TrialStep[]
  provenance?: ContentProvenance
}
```

Storage: add `trialWorkflows?: TrialWorkflow[]` to `Playbook`, so it lives in
`data/playbook.json` + Firestore and propagates live like everything else.

## Rendering

- **One** generic component, e.g. `TrialWorkflowRenderer`, renders any
  `TrialWorkflow` using existing card/checklist/link patterns from
  [UI_RULES.md](./UI_RULES.md). No per-workflow bespoke components.
- New presentation needs ⇒ add a new typed field/block type to the schema + a
  branch in the renderer (a reviewed code change), not generated markup.

## Where it lives in the flow

The trial flow is a post-Close / enablement module (a trial starts after a
positive call). It is **not** shoehorned into the 5 call stages. Confirm
placement against [UX_FLOW.md](./UX_FLOW.md) when implementing; adding a screen
or module is a structural change to record there.

## Authoring (reuses Admin Generation)

A trial workflow is generated by the same pipeline as content
([ADMIN_GENERATION.md](./ADMIN_GENERATION.md)) with `target: "trialWorkflow"`:

1. Admin links a **Google Doc** (the trial SOP) — or another source.
2. Generate → AI parses the doc into a draft `TrialWorkflow` (schema-valid).
3. Review the diff, amend steps/links/language.
4. Save → live to reps via Firestore sync.

So the Google Doc is just another **source**; the trial workflow is just another
**generated, reviewed, frozen** artifact. Same stable-ID/merge rules apply.

## Guardrails
- AI output is a `TrialWorkflow` (data) only — never components, never code.
- Links are reviewed by the admin before save (don't blindly trust generated
  URLs).
- The renderer is fixed and reviewed; it is the only thing that turns config
  into UI.
